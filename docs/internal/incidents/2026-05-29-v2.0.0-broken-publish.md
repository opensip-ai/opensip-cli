# Incident: v2.0.0 shipped a broken publish (missing cli-ui export)

**Date:** 2026-05-28 → 2026-05-29
**Severity:** High — every fresh install of `@opensip-tools/cli@2.0.0` crashed on startup.
**Resolution:** Republished as `2.0.1`; deprecated the broken `2.0.0` artifacts.

## Symptom

Any `opensip-tools` invocation under `2.0.0` (including `--version`) crashed
immediately:

```
SyntaxError: The requested module '@opensip-tools/cli-ui' does not
provide an export named 'RunFooterHints'
  at .../@opensip-tools/fitness/dist/cli/fit-runner.js:25
```

## Root cause

The 2.0.0 release was **non-atomic**, and npm version immutability froze the
inconsistency in place:

- `@opensip-tools/cli-ui@2.0.0` was published at **2026-05-28T18:32Z**, from a
  build whose `dist/` predated the `RunFooterHints` export — the symbol (and its
  source file `run-footer-hints.tsx`) was absent from the published tarball.
- The rest of the set — `core`, `contracts`, `fitness`, `simulation`, `graph`,
  `cli` — published **~11 hours later** at **2026-05-29T05:47–05:56Z**, after
  several fix commits (`vitest` CI timeout, graph tarball-glob). `fitness@2.0.0`
  in that batch **imports `RunFooterHints` from cli-ui**.
- Because `pnpm pack` resolves `workspace:*` to an **exact** version,
  `fitness@2.0.0` and `cli@2.0.0` depend on `cli-ui@2.0.0` *exactly* — and that
  exact version was the stale one. npm refused to overwrite the already-published
  `cli-ui@2.0.0`, so the re-run advanced everything *except* cli-ui.

Net: a dependency graph where every version string says `2.0.0` and every range
resolves, but the compiled bytes of cli-ui don't provide a symbol fitness imports.

## Why existing gates missed it

`scripts/verify-release.mjs` validates version *strings*, tag agreement, the
CHANGELOG entry, doc sync, and cross-package dependency *ranges*. None of those
can detect "package A's compiled output doesn't export what package B imports."
That is only observable by **loading the packed artifacts together**, which
nothing did before publishing.

The `pnpm -r run clean` guard in `release.yml` was real but orthogonal — it
prevents *stale orphan files leaking in*, not a *whole package frozen at an
earlier version* by a partial publish.

## Fix

1. Bumped all 27 `@opensip-tools/*` packages to `2.0.1` and republished the
   complete set from one clean build (`CHANGELOG.md` → `## [2.0.1]`).
2. Deprecated the broken artifacts so the registry warns installers:
   - `npm deprecate '@opensip-tools/cli@2.0.0' '…use 2.0.1'`
   - `npm deprecate '@opensip-tools/cli-ui@2.0.0' '…superseded by 2.0.1'`

## Prevention (systemic)

Added a **post-pack / pre-publish smoke test** to the release workflow:
`scripts/smoke-pack.mjs` (run as `pnpm smoke-pack`, wired into `release.yml`
between the pack and publish steps).

It installs the freshly-packed tarballs *together* into a throwaway consumer
project — forcing every `@opensip-tools/*` dependency to its local tarball via
npm `overrides`, since the about-to-publish versions aren't on the registry yet
— then loads the CLI (`--version`, then `--help`). An inter-package export/ABI
mismatch fails the step **before any `npm publish`**, where version immutability
would otherwise make it unrecoverable.

Validated against this incident: pointing the smoke test at the real broken
`cli-ui@2.0.0` tarball reproduces the `RunFooterHints` SyntaxError and exits
non-zero; the good `2.0.1` set passes.

## Standing follow-ups

- **2FA / authenticated writes for the `opensip` npm account.** The deprecate
  step initially failed (`EOTP`) and was unblocked with an automation token in
  `~/.npmrc`. Decide whether to keep that token (a standing on-disk secret) or
  set up an authenticator so manual writes (deprecate, dist-tag, access changes)
  work without it.
- **Consider making the publish itself harder to leave partial** — e.g. abort
  the run if any target version already exists on npm at the *start* of a fresh
  release (as opposed to the current idempotent per-package skip, which is right
  for resuming a same-version retry but masks a partial-publish-then-edit).
