# ADR-0097: Gate verdict determinism

```yaml

title: Gate verdict determinism
date: 2026-06-29
status: active
supersedes: []
superseded_by: null
related: [ADR-0011, ADR-0020, ADR-0036, ADR-0065, ADR-0075, ADR-0094]
tags: [gates, json, provenance, sessions, agents]
enforcement: mechanizable
enforcement-reason: >
  Declared-input stamping is host-owned in the envelope delivery/output seams,
  and tests assert the manifest is additive, deterministic, and limited to an
  explicit allowlist. Gate/verdict tests continue to assert one envelope verdict
  drives the process exit.
```

**Decision:** A persisted or emitted gate verdict must carry a host-stamped,
allowlisted `declaredInputs` manifest that explains the runtime inputs needed to
diagnose the verdict. The manifest is additive JSON metadata on
`SignalEnvelope`; it must never include environment dumps, absolute project paths,
secrets, or mutable hidden host state.

The current manifest records:

- CLI package version.
- Node version.
- Package manager identity from nearest `package.json#packageManager`, falling
  back to `npm_config_user_agent`.
- Platform (`process.platform/process.arch`).
- Tool id.
- Tool/engine package version when available from the active tool manifest.
- Baseline fingerprint identity from the envelope.

**Alternatives:**

- **No manifest; rely on logs.** Rejected. Logs are useful for debugging, but
  JSON/SARIF/session consumers need provenance attached to the machine artifact
  they are already reading.
- **Dump all environment variables and config.** Rejected. That would leak
  credentials and make verdict payloads nondeterministic.
- **Make tools self-report the manifest.** Rejected. Tools would drift and could
  forget fields; host stamping keeps the surface uniform for first-party and
  extension tools.
- **Make `declaredInputs` required immediately.** Rejected. It is additive while
  the ecosystem is pre-1.0 and older producers may still omit it.

**Rationale:** Gate failures are often caused by a change in more than source
code: CLI version, package manager version, tool engine version, Node version, or
baseline fingerprint identity can all change the result. Agents and CI systems
need a compact, explicit manifest so they can compare "same code, different
inputs" without guessing from logs or host-local state.

The manifest is intentionally small. It records declared runtime facts that can
explain verdict skew, not every possible input to static analysis. Future fields
must meet the same bar: useful for verdict diagnosis, safe to publish in JSON,
and generated by the host from an explicit allowlist.

**Consequences:**

- `SignalEnvelope.declaredInputs` is optional for compatibility, but new host
  output paths stamp it before JSON outcome rendering, cloud sync, SARIF
  delivery, and dashboard/report composition.
- Consumers should treat absence as "older/no-manifest producer", not as proof
  that the inputs are default.
- Docs and dashboard surfaces should show the manifest compactly so humans and
  agents can compare runs.
- Future gate hardening must preserve one canonical verdict: the envelope verdict
  remains the source for exit code, session history, dashboard status, and egress.

**Fitness check:** No standalone check yet. The host-owned stamping is covered by
unit tests; broader gate determinism remains enforced by the existing
CommandOutcome, baseline identity, and gate verdict tests.
